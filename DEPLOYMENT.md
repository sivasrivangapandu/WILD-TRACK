# WildTrackAI Deployment Guide

## 🚀 Deployed URLs

- **Backend API**: https://wildtrack-backend-j9n8.onrender.com
- **Frontend App**: https://wildtrack-frontend.onrender.com
- **Health Check**: https://wildtrack-backend-j9n8.onrender.com/health

## 📋 Deployment Architecture

```
GitHub Repository
    ↓
Render Blueprint (render.yaml)
    ├─→ Backend (FastAPI)
    │   - Free web service
    │   - Auto-downloads .h5 models from GitHub Release
    │   - Python 3.12, SQLite database
    │
    └─→ Frontend (React + Vite)
        - Static site
        - Connects to backend via VITE_API_URL
```

## 🔧 Setup Instructions

### 1. Environment Variables

#### Backend (Render Dashboard)
Set these in your Render backend service:
- `GEMINI_API_KEY` - Your Google Gemini API key
- `NINJA_API_KEY` - API Ninjas API key
- `JWT_SECRET` - Auto-generated by Render

#### Frontend (Render Dashboard)
Set this in your Render frontend service:
- `VITE_API_URL` - Your backend URL (e.g., `https://wildtrack-backend-j9n8.onrender.com`)

### 2. Model Files

Model files are automatically downloaded from the GitHub Release at startup:
- Release: https://github.com/sivasrivangapandu/WILD-TRACK/releases/tag/v1.0-models
- Files: `wildtrack_complete_model.h5`, `wildtrack_final.h5`
- Total size: ~82 MB

⚠️ **First startup takes 2-3 minutes** to download models.

### 3. Database

SQLite database (`wildtrack.db`) is automatically created on first run.
- Stores: predictions, chat sessions, user accounts
- Persists on Render's ephemeral disk (lost when instance spins down on free tier)

## 🔄 Deployment Workflow

### Automatic Deployment
```bash
# Any push to main branch triggers auto-deploy
git push origin main
```

Render will:
1. Detect changes via Blueprint sync
2. Build backend: install requirements.txt
3. Build frontend: npm install && npm run build
4. Deploy both services

### Manual Redeploy
From Render Dashboard:
1. Select the service
2. Click "Manual Deploy" → "Deploy latest commit"

## ⚠️ Known Limitations (Free Tier)

1. **Cold Starts**: Services spin down after 15 min inactivity
   - First request after spin-down: ~30-60s (model download + startup)
   - Subsequent requests: < 1s

2. **Storage**: Ephemeral disk - database resets on restart
   - Use external database (e.g., Render Postgres) for persistence

3. **Bandwidth**: 100 GB/month shared across services

4. **Build Time**: ~3-5 minutes per deploy

## 🐛 Troubleshooting

### Backend fails to start
Check logs for:
- Missing dependencies → verify `requirements.txt`
- Model download errors → check GitHub Release accessibility
- Port binding errors → Render uses `$PORT` env var

### Frontend shows "Network Error"
- Verify `VITE_API_URL` is set correctly
- Check CORS headers in backend (allows all origins by default)
- Confirm backend is running (check health endpoint)

### Models not loading
- Check `/health` endpoint for `model_download_status`
- Verify GitHub Release assets are public
- Check disk space availability

## 🔐 Security Notes

1. **Secrets**: Never commit `.env` files or API keys
2. **Database**: Default setup uses SQLite (embedded)
   - For production: migrate to Postgres
3. **CORS**: Currently allows all origins
   - For production: restrict to frontend domain only

## 📦 Updates & Maintenance

### Update Dependencies
```bash
# Backend
cd backend
pip install <new-package>
pip freeze > requirements.txt
git push

# Frontend
cd frontend
npm install <new-package>
git push
```

### Update Models
1. Train new model locally → generate .h5 file
2. Upload to GitHub Release (use existing v1.0-models or create new tag)
3. Update `MODEL_URLS` in `backend/main.py`
4. Push changes

### Database Migrations
```bash
# Create new migration
cd backend
# Add new models to models/
# Update database.py if needed
git push
# Render will auto-recreate schema
```

## 🎯 Performance Tips

1. **Prediction Optimization**
   - Enable TTA for higher accuracy (slower)
   - Disable GradCAM for faster responses
   - Use batch endpoint for multiple images

2. **Frontend Optimization**
   - Already includes PWA support
   - Static assets cached automatically
   - Use WebP images where possible

3. **Database**
   - Default SQLite is fine for < 1000 predictions/day
   - For higher traffic: switch to Postgres
